## 评估动画机会

分析动效能在何处改善体验：

1. **找出偏静态的区域**：
   - **缺少反馈**：操作后没有视觉确认（按钮点击、表单提交等）  
   - **过渡生硬**：状态瞬间切换显得突兀（显隐、首屏、路由切换）  
   - **关系不清**：空间或层级关系不直观  
   - **缺少惊喜**：能用但缺少愉悦感  
   - **引导不足**：本可引导注意力或解释行为的时机  

2. **理解情境**：
   - 产品气质？（活泼 vs 严肃，热烈 vs 克制）  
   - 性能预算？（移动优先？重页面？）  
   - 受众是谁？（对动效敏感的用户？追求速度的资深用户？）  
   - 优先级？（一个主视觉动画 vs 大量微交互？）  

若从代码库无法推断以上任一点，**直接向用户提问**。

**关键：** 遵守 `prefers-reduced-motion`。必须为需要减少动效的用户提供**非动画或等价替代**。

## 制定动画策略

做一份**有目的**的动画计划：

- **高光时刻**：**唯一**的标志性动画是什么？（首屏加载？Hero？关键交互？）  
- **反馈层**：哪些交互需要「被看见」的回应？  
- **过渡层**：哪些状态变化需要顺滑衔接？  
- **惊喜层**：哪里可以适度出人意料？  

**重要：** 一处编排精良的体验，胜过到处撒零碎动画。把力气花在**高影响时刻**。

## 实现动画

按类别系统性地加入动效：

### 入场动画
- **首屏编排**：元素错开出现（间隔约 100–150ms）、淡入 + 位移组合  
- **Hero 区**：主内容更有记忆点的入场（缩放、视差或其它创意）  
- **内容显现**：用 Intersection Observer 做滚动触发  
- **模态/抽屉**：滑入 + 淡入、背景淡入、焦点管理  

### 微交互
- **按钮**：
  - 悬停：轻微缩放（1.02–1.05）、颜色过渡、阴影加重  
  - 点击：快速按下再弹起（0.95 → 1）、涟漪等  
  - 加载：转圈或脉冲态  
- **表单**：
  - 聚焦：描边色过渡、轻微缩放或光晕  
  - 校验：错误时轻晃、成功时打勾、颜色平滑过渡  
- **开关**：滑动 + 颜色（约 200–300ms）  
- **复选/单选**：勾选动画、涟漪  
- **点赞/收藏**：缩放 + 旋转、粒子感、颜色过渡  

### 状态过渡
- **显隐**：淡入淡出 + 位移（不要瞬间切换），时长约 200–300ms  
- **展开/收起**：高度过渡并处理好 `overflow`，图标旋转  
- **加载**：骨架屏淡入、转圈、进度条  
- **成功/失败**：颜色过渡、图标动画、轻微缩放脉冲  
- **启用/禁用**：透明度、指针样式  

### 导航与流程
- **页面切换**：路由间交叉淡入、共享元素过渡  
- **Tab**：指示条滑动、内容淡入/滑移  
- **轮播**：平滑 `transform`、吸附点、惯性滚动  
- **滚动**：视差层、粘性头状态变化、滚动进度指示  

### 反馈与引导
- **悬停提示**：Tooltip 淡入、指针变化、高亮  
- **拖拽**：抬起感（阴影 + 缩放）、放置区高亮、平滑落位  
- **复制/粘贴**：粘贴后短暂高亮、「已复制」反馈  
- **焦点路径**：在表单或流程中突出焦点走向  

### 惊喜时刻
- **空状态**：插图轻微浮动  
- **完成操作**：纸屑、对勾动画、轻量庆祝  
- **彩蛋**：可发现的隐藏交互  
- **情境动效**：天气、时段、季节等弱主题点缀  

## 技术实现

按场景选用合适手段：

### 时长与缓动

**按用途选时长：**
- **100–150ms**：即时反馈（按下、开关）  
- **200–300ms**：状态变化（悬停、菜单展开）  
- **300–500ms**：布局变化（手风琴、模态）  
- **500–800ms**：入场（首屏编排）  

**缓动曲线（建议用下列，少用浏览器默认）：**
```css
/* 推荐 — 自然减速 */
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);    /* 顺滑、克制 */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);   /* 略干脆 */
--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);     /* 利落、果断 */

/* 避免 — 易显过时、廉价 */
/* bounce: cubic-bezier(0.34, 1.56, 0.64, 1); */
/* elastic: cubic-bezier(0.68, -0.6, 0.32, 1.6); */
```

**退出动画应比入场更快**，约为入场时长的 **~75%**。

### CSS 动画
```css
/* 简单、声明式动效优先 */
- 状态变化用 transition
- 复杂序列用 @keyframes
- 仅用 transform + opacity（易 GPU 加速）
```

### JavaScript 动画
```javascript
/* 复杂、需交互控制的动效 */
- 程序化控制用 Web Animations API
- React 可用 Framer Motion
- 复杂时间轴可用 GSAP
```

### 性能
- **GPU**：优先 `transform` 与 `opacity`，少动会触发布局的属性  
- **will-change**：仅对已知昂贵动画少量使用  
- **减少重绘**：控制 repaint，必要时用 `contain`  
- **帧率**：目标设备上尽量稳定 60fps  

### 无障碍
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

**禁止：**
- 使用 bounce / elastic 类缓动 —— 易显过时，且把注意力拉到「动画本身」  
- 动画化布局属性（width、height、top、left）—— 改用 `transform`  
- 交互反馈超过约 500ms —— 易感觉拖沓  
- 无目的的动画 —— 每一段动效都应有理由  
- 忽略 `prefers-reduced-motion` —— 属于无障碍违规  
- 万物皆动 —— 动效疲劳会让界面显得累  
- 除非刻意设计，否则不要在动画期间阻塞操作  

## 质量验证

充分自测动效：

- **约 60fps 顺滑**：目标设备上无明显卡顿  
- **体感自然**：缓动有机，不「机械」  
- **时长合适**：既不太快（吓人）也不太慢（拖沓）  
- **减少动效生效**：关闭或简化得当  
- **不挡操作**：动效进行中/结束后仍可交互  
- **有增益**：让界面更清晰或更愉悦  

请记住：动效应**帮助理解与反馈**，而不只是装饰。**有目的地**做动画，尊重性能与无障碍。**好的动效往往不被注意**——只是让一切「感觉对了」。
