# 本地草稿与断网保护 

## 当前实现摘要


| 层级   | 现状                                                                                                                                                                                                                                                                                                   |
| ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 前端   | `[frontend/app/(app)/dreams/new/page.tsx](frontend/app/(app)`/dreams/new/page.tsx) 中 `DreamEditor` 将标题、正文、情绪、睡眠等大量状态放在 React state，提交时组装 `[CreateDreamPayload](frontend/lib/dream-api.ts)` 并调用 `DreamApi.create` → `POST /dreams`（编辑模式为 `PATCH /dreams/{id}`）。**无** localStorage / IndexedDB / 离线队列。 |
| 后端   | `[backend/app/api/dreams.py](backend/app/api/dreams.py)` 仅有 `POST ""` 创建；**不存在** `/quick-capture` 路由。                                                                                                                                                                                                |
| 草稿语义 | `[DreamService.create_dream](backend/app/services/dream_service.py)` 中 `is_draft=request.title is None`：不传 `title` 即服务端标记为草稿。                                                                                                                                                                        |


因此：「网络恢复后同步到 `/quick-capture`」属于**规划中的接口**，与现网代码不一致。落地时有两种等价策略（见下文「同步目标」）。

---

## 业界常见做法（与产品对标）

- **Gmail / Google Docs**：高频自动保存 + 服务端版本；离线时本地队列，恢复后重放；冲突策略多为「服务端优先 + 合并提示」。
- **Notion / Figma**：IndexedDB 存大块状态，同步层与 UI 解耦；乐观 UI + 后台重试。
- **Medium / 论坛发帖**：localStorage 或 sessionStorage 存轻量草稿；提交失败保留草稿并提示重试。
- **共同点**：**防抖写入本地**、**页面卸载前强制 flush**、**在线事件触发重试**、**成功后清理本地草稿**、**敏感内容不写入可共享的 analytics**。

对「醒来网络差」场景，**P0 是「不丢字」**（本地可靠持久化），**P1 是「恢复后自动或一键同步」**。

---

## 存储选型：localStorage vs IndexedDB


| 维度     | localStorage               | IndexedDB（推荐主存）           |
| ------ | -------------------------- | ------------------------- |
| 容量     | 约 5MB，各浏览器不同               | 通常大得多，适合长文 + 多字段 JSON     |
| 同步 API | 阻塞主线程，大 JSON 序列化可能卡顿       | 异步，适合 debounce 后批量写入      |
| 本表单    | `DreamEditor` 字段多 + 正文可能很长 | 单条草稿 JSON 很容易超过几百 KB 长期积累 |


**建议**：以 **IndexedDB** 作为「梦境编辑草稿」的权威本地存储

实现上可用轻量封装：`**idb-keyval`** 或 **Dexie.js**（若未来有多草稿、多表再引入、现阶段只是单条草稿）。

## 草稿数据模型（建议）

单键即可满足「新建页一条草稿」；若未来支持「多条未发布草稿」，再改为 `draftId` 列表(暂不需要支持多条未发布草稿)。

```ts
// 概念结构（非现网代码）
type DreamLocalDraft = {
  schemaVersion: number;           // 迁移用，例如 1
  updatedAt: number;               // Date.now()
  route: "create" | "edit";
  dreamId?: string;                // 编辑模式时有
  form: Record<string, unknown>;   // 与 DreamEditor 可序列化字段对齐，或显式列出所有字段
  pendingSync?: boolean;           // 是否曾提交失败待重试
  lastErrorAt?: number;
};
```

- **序列化**：只存可 JSON 化的字段；**图片/录音**若仍为 `File`/`Blob`，需 **IndexedDB 存 Blob**，或仅存「待上传列表引用」（见下文附件）。
- **schemaVersion**：以后加字段时可做一次性迁移或丢弃旧草稿并提示。

---

## 何时写入本地（防抖 + 生命周期）

1. **防抖**：内容变化后 **300ms～2s** 写入 IDB（正文可略长防抖，避免每键入都写）。
2. **强制 flush**：`visibilitychange`（切后台）、`pagehide`、`beforeunload` 时再写一次（`beforeunload` 中避免重逻辑，只做同步写或 `navigator.sendBeacon` 不适用存 IDB，故以 **pagehide + visibility** 为主）。
3. **首次进入创建页**：从 IDB 读取；若存在 `updatedAt` 与当前空表单冲突，采用 **「恢复草稿」确认**（避免覆盖用户有意清空）。

## 断网与同步策略

### 检测

- `navigator.onLine` 仅作粗判；真实可用性以 `**DreamApi.create` 成功/失败** 为准。
- 监听 `window` 的 `**online`** 事件：若存在 `pendingSync === true`，触发 **重试队列**（见下）。

### 同步目标（与 `/quick-capture` 的关系）

- **现状**：无 `/quick-capture`，应二选一：
  - **A（推荐，改动小）**：恢复在线后仍调用现有 `**POST /dreams`**，payload 与线上一致；若希望「先占坑草稿」，可 `**title` 省略** 以利用现有 `is_draft=True` 行为，再 `PATCH` 补全字段（需产品确认「草稿是否出现在列表」）。
  - **B（产品化 quick capture）**：新增 `**POST /dreams/quick-capture`**（仅 `content`、`dream_date`、可选 `client_draft_id`），返回 `id` 后再 `PATCH` 丰富字段；与规划文档一致，但需后端新增路由与校验。

**离线队列**建议结构：

```text
队列项: { id: uuid, payload: CreateDreamPayload, retries: number, createdAt }
```

- 成功：`invalidateCache` 已有、清除本地草稿、从队列移除。
- 失败：指数退避重试（如 1s/2s/4s，上限），**401** 单独处理（token 失效则暂停队列并引导登录）。

### 与「仅本地草稿」的边界

- **未登录**：仍可写 IDB；登录后再入队同步（若你的业务允许匿名编辑）。
- **已登录离线**：只保证本地不丢；**上线后**再 `POST`。

## 附件与语音（与断网强相关）

当前提交流程在创建成功后再 `[getAttachmentUploadUrl` + OSS](frontend/app/(app)/dreams/new/page.tsx) 上传图片。断网时：

- **图片**：Blob 存 IndexedDB，同步成功拿到 `dream.id` 后再上传；失败保留「待上传」标记并重试。
- **语音**：若 `[VoiceRecorder](frontend/components/dream/voice-recorder.tsx)` 产出的是已上传 URL 或 base64，需统一：**本地先存 Blob/文件句柄**，联网后再走现有转写/上传接口（暂不需要支持语音附件）。

## 多标签页与隐私

- **多标签**：`BroadcastChannel` 或 `storage` 事件同步「最后更新时间」；冲突策略用 **LWW（last write wins）** 或提示「另一窗口已更新草稿」。
- **隐私**：草稿含敏感梦境内容；**不要**发往第三方分析；IndexedDB 同源隔离；若未来有「导出」需求再考虑可选加密（一般产品阶段不必先做）。

---

## 成功后的清理

- `handleSubmit` 成功且拿到 `dream.id` 后：**删除**对应 IDB 记录与队列项，避免用户再次进入看到旧草稿。

## 与现有代码的衔接点（实施时）

- 在 `[DreamEditor](frontend/app/(app)`/dreams/new/page.tsx) 中抽取 **「可序列化的表单快照」** 与 `**hydrateFromSnapshot`**，避免手写几十个 `setState` 两遍。
- 新建小模块：例如 `[frontend/lib/dream-draft-storage.ts](frontend/lib/dream-draft-storage.ts)`（IDB + 防抖 + 恢复）与可选 `[frontend/hooks/use-dream-draft-sync.ts](frontend/hooks/use-dream-draft-sync.ts)`（online + 队列）。
- 后端：若坚持 `/quick-capture`，在 `[backend/app/api/dreams.py](backend/app/api/dreams.py)` 增加路由并复用 `DreamService` 最小创建逻辑；否则 **无需改后端** 即可先做「纯本地草稿 + 在线 `POST /dreams`」。

## 小结


| 目标        | 做法                                             |
| --------- | ---------------------------------------------- |
| 不丢内容      | IndexedDB 存完整表单快照 + 生命周期 flush                 |
| 弱网        | 失败入队 + `online` 重试 + 退避                        |
| 与现 API 对齐 | 优先 `POST /dreams`；`/quick-capture` 为增强项需新接口    |
| 服务端草稿     | 可利用现有 `title is None` → `is_draft`，需定义列表是否展示草稿 |


以上即为与当前仓库一致、且贴近主流产品行为的「本地草稿与断网保护」最佳实践方案；**不要求**用户必须新增 `/quick-capture` 才能上线本地草稿能力。
